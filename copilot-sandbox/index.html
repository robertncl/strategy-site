What holds when autopilot is on?

GitHub Copilot CLI's --autopilot --allow-all-tools removes the human from the loop. The question isn't whether to trust the model — it's what boundary holds when the agent's own approvals are turned off.

What we're controlling

Copilot CLI's autonomy flags remove every guardrail.

With --autopilot --allow-all-tools the agent runs shell commands, writes files, and calls MCP tools with zero approval prompts until the task finishes. Tool-approval and sandboxing are two different switches — turning off approvals does not by itself enable the sandbox.

0approval prompts in full autopilot
2separate switches: tool-approval vs. sandbox
∞commands the agent can run unchecked

A sufficiently capable autopilot session — or a prompt injection — that can edit ~/.copilot/settings.json can weaken its own cage. That is the gap the out-of-process solutions close.

What a runaway agent attempts

The probe exercises every escape path.

  • Read host secrets — ~/.ssh/id_rsa, /etc/shadow, ~/.aws
  • Write outside the workspace — /usr, system paths
  • Reach non-approved hosts — example.com
  • POST to a read-only API (Layer-7 method escape)
  • Connect to an unapproved MCP server

Every candidate is graded on the same eight checks.

Host isolation, filesystem control, network allow/block, Layer-7 method filtering, and MCP allow/block — if the sandbox fails any check, the autonomous agent escapes that boundary.

The candidates

Three solutions at three different altitudes.

Each sits at a different point relative to the agent's process — and that single fact drives everything about its strength.

Inside the agent

Copilot Local Sandbox

Microsoft MXC — runs each tool call through a local sandbox governed by settings.json. Finest MCP control (per-tool), but same trust domain as the agent.

Outside · container

Docker Sandbox

Copilot runs inside a hardened container — read-only rootfs, cap-drop ALL, egress proxy. Independent of any agent flag. Portable and proven.

Outside · kernel + proxy

NVIDIA OpenShell

Kernel LSM + policy proxy. Agent believes it has full autonomy; every syscall and connection is judged against a YAML policy the agent cannot read or modify.

Solution A · Inside the agent

Copilot Local Sandbox (MXC)

Even with --allow-all-tools, when sandbox.enabled is true the MXC layer still denies reads of ~/.ssh, blocks non-listed hosts, and refuses writes outside the workspace. MCP is gated at the finest granularity of any solution — per tool.

Filesystem rules

Per-path access: read-write for workspace, deny for secret paths. Default-deny network with host-level rules.

Per-tool MCP

--deny-tool always wins over any allow. The only solution that gates individual MCP tools.

Cross-platform

Consistent across macOS, Linux and Windows. Lowest setup cost — a JSON config file.

Limitation: same trust domain. The policy is user-writable; a prompt injection that edits settings.json can weaken the cage for the next run. No L7 method filtering. Lock the policy with Intune/MDM on managed devices.

Solution B · Outside · container

Docker Sandbox

Copilot runs with full autonomy inside a hardened container. The boundary is independent of any agent flag — it cannot be weakened from within the session.

Read-only rootfs

Writable tmpfs + a single workspace volume. The agent can only write where you let it.

No privileges

cap_drop: ALL, no-new-privileges, non-root UID. No escalation regardless of --yolo.

Egress proxy

Internal network + tinyproxy default-deny allow-list. Approved hosts reachable, everything else blocked.

Secrets never mounted

Host secrets are simply never present in the container — isolation by absence.

MCP by reachability

Approved MCP server on the internal net; unapproved server only on the egress net — unreachable.

Portable

Runs anywhere Docker runs. Familiar tooling for CI and dev environments.

Limitation: no L7 method filtering without TLS MITM. MCP control is coarse (reachable or not). No credential proxy — secrets in env/volumes are visible to the agent.

Solution C · Outside · kernel + proxy

NVIDIA OpenShell

Copilot runs unmodified in full autopilot and believes it has full autonomy. Every syscall and connection is judged against a declarative YAML policy the agent cannot read or modify.

Landlock LSM

Kernel-level filesystem enforcement. Denies writes to read-only paths even at UID 0 inside the namespace.

L7 method filtering

Terminates TLS and filters HTTP methods natively. GET allowed, POST/PUT/DELETE blocked. The only solution that passes the L7 check.

Credential proxy

Real keys replaced by opaque placeholders. The proxy injects the real key upstream and fails closed on error.

Seccomp + namespaces

Process isolation via seccomp-BPF, mount namespaces, and non-root enforcement. Host paths are invisible.

Inference routing

Org policy can reroute sensitive prompts to a local model regardless of what the agent requested.

Full audit trail

Gateway logs every allow/deny decision. Streams in real time for incident response.

Limitation: MCP is connection-level, not per-tool. Linux only (kernel LSM). Needs an OpenShell gateway + Docker daemon. Pair with Copilot --deny-tool for tool-level control.

They're complementary

Layer all three for maximum assurance.

The three solutions are defense-in-depth, not competitors. Each adds a capability the others lack.

  1. 1

    Always leave the Copilot local sandbox enabled

    It adds the one thing the outer layers lack — per-tool MCP gating. --deny-tool beats any allow. Lock the policy with Intune/MDM.

  2. 2

    Use Docker where there's no OpenShell gateway

    Dev boxes, generic CI runners. Strong portable host isolation and egress allow-listing. Accept that L7 and per-tool MCP are out of scope.

  3. 3

    Use OpenShell for production & real credentials

    The only out-of-process boundary with L7 method filtering, fail-closed credential proxy, inference routing, and a full audit trail.

  4. 4

    Always cap the loop

    --max-autopilot-continues N — none of the sandboxes bound agent steps or token spend. Scope writable paths tightly.

Recommendation

Implement OpenShell as the security boundary.

Keep the Copilot local sandbox enabled as an inner layer. Use Docker where you need portable isolation without an OpenShell gateway. The three are complementary — defense in depth, not a choice between them.
