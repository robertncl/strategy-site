The shift

AI stopped suggesting. It started executing.

Agents now run shell commands, execute generated Python, transform spreadsheets, and open pull requests. That code is, by definition, untrusted — written by a model, on the fly, against your data. You can't run it on the bare metal that holds production.

100%of agentic actions execute model-generated code or tool calls
3distinct sandbox surfaces in the GitHub + Microsoft stack
1shared isolation substrate underneath them all

A sandbox is the contract that lets you say yes to autonomy: an isolated, ephemeral environment where AI can act freely and still touch nothing it shouldn't.

The problem

Autonomy without isolation is a breach waiting to happen.

  • Generated code can read secrets, env vars, and the wider filesystem
  • An agent with network egress can exfiltrate data or pull malware
  • Prompt injection turns a helpful tool into an attacker's hands
  • Long-lived runtimes accumulate state — and blast radius

You want the productivity of autonomous agents without granting them the keys to production.

The answer everywhere is the same shape: ephemeral, isolated, network-restricted compute that is created for one task and destroyed after.

The landscape

Three products. Three audiences. One idea.

Microsoft and GitHub expose sandboxing at three different altitudes — matched to who is building and what they're running. Same principle, different surface.

For developers

GitHub Copilot

Local & cloud sandboxes that contain a coding agent as it runs shell commands and edits repos — on your machine or in an ephemeral cloud box.

For makers

Copilot Studio

A built-in code interpreter that lets low-code business agents run Python on files inside a locked-down, monitored VM.

For builders & pros

Azure AI Foundry

Code-first agents with a sandboxed Python tool and per-session VM-isolated hosted agents, wired into your Azure project.

All three converge on the same plumbing — covered on the architecture slide.

Option 1 · For developers

GitHub Copilot — local & cloud sandboxes.

In public preview (June 2026), Copilot can run inside an isolated sandbox so the agent can touch code, tools, the filesystem, and the network safely — either contained on your own machine or in a throwaway cloud box.

Local sandbox

On your machine

Enable with /sandbox enable in the Copilot CLI. Restricts the agent's access to filesystem, network, and system capabilities — macOS, Linux & Windows.

  • No extra cost on Copilot seats
  • Policy via Intune / MDM
  • Keeps execution on-box

Cloud sandbox

Ephemeral Linux

Launch with copilot --cloud for a fully isolated, ephemeral Linux box hosted by GitHub. Sessions are Active, Stopped, or Deleted — resume across devices.

  • Built on Azure Container Apps
  • Parallel, off-machine compute
  • Inherits cloud-agent policy

Coding agent

PR-gated

The coding agent works in an ephemeral environment powered by GitHub Actions — one container per task. Nothing merges without human review.

  • Per-task isolation
  • Human approval to merge
  • Self-hosted runners via ARC

GitHub owns the identity, policy, and billing layer; the isolation runs on Azure. Org admins gate access with the Cloud Sandbox access policy.

Option 2 · For makers

Copilot Studio — the code interpreter sandbox.

A Python execution engine built into Copilot Studio agents and prompt builder. Low-code makers get real data analysis and file processing — Excel, Word, PowerPoint, PDF — without ever leaving the Power Platform, and without the agent touching anything outside the session.

Isolated VMs

Runs in Microsoft-hosted, isolated virtual machines on Azure; each session is separated from others and from core services.

No network

Strict constraints block external network access, system commands, and unauthorized file operations by design.

Live-monitored

All executed code is scanned in real time; suspicious activity terminates the session and alerts security teams.

Ephemeral

Compute is created for the session and disposed of afterward — no data persistence between runs.

Scoped data

Only sees the files and inputs handed to it in the prompt — no broader tenant data or user context.

Admin-gated

Off by default; a tenant admin enables it in the Power Platform admin center, with environment-level control.

Compliant to SOC 2, ISO 27001 & GDPR; your data and executed code are never used to train models. Public cloud only today — no GCC / GCC High yet.

Option 3 · For builders & pros

Azure AI Foundry — code interpreter & hosted agents.

Code-first agents get a sandboxed Python tool: the Foundry model writes and runs code iteratively for data analysis, math, and chart generation, with files uploaded in and artifacts downloaded out. Available across the Python, C#, JavaScript, Java & REST SDKs.

Dynamic sessions

Built on Azure Container Apps dynamic sessions — purpose-built for running untrusted code at scale.

Hyper-V isolation

Each session is isolated by a hardware-level Hyper-V boundary; concurrent conversations get separate sessions.

No egress

The sandbox can't make outbound network requests and doesn't inherit your agent's subnet configuration.

Region-pinned

Runs in the same Azure region as your Foundry project — predictable data residency.

Hosted agents

Deployed agents run in per-session VM-isolated sandboxes, reaching tools via a managed Toolbox MCP endpoint.

Custom control

Need custom packages or a VNet? The custom code interpreter gives you control over the runtime & isolation model.

Sessions stay active up to 1 hour with a 30-minute idle timeout, and bill on top of token usage. Availability varies by region & model.

Why it's safe

Defense in depth, by construction.

A sandbox isn't one control — it's a stack of them. The same layers show up, in different combinations, across all three offerings.

Hardware isolation

Hyper-V / VM boundaries separate each session from other sessions and from the host platform.

Ephemerality

Created per task, destroyed after — there is nothing persistent for an attacker to land on.

No / gated egress

Outbound network is denied or policy-controlled, cutting exfiltration and malware-pull paths.

Least-privilege data

The runtime sees only the inputs explicitly attached — not tenant data or ambient secrets.

Runtime monitoring

Executed code is scanned for malicious patterns; suspicious sessions are killed and alerted.

Human gates

Coding-agent changes need review before merge; admins must opt the capability on.

Who controls it

Enterprise controls, not a free-for-all.

  • Copilot cloud sandboxes gated by an org access policy; local enforced via Intune/MDM
  • Copilot Studio code interpreter is off by default — tenant admin opts in per environment
  • Foundry sandboxes inherit Azure RBAC, region pinning, and project-level controls
  • Execution traces, prompts, and outputs are logged for audit across the board

Compliance is inherited, not rebuilt.

Microsoft-hosted sandboxes carry SOC 2, ISO 27001 and GDPR commitments, and your data and executed code are not used to train models — so adoption rides on existing assurances rather than a fresh review.

The governance question becomes "which environment, for whom?" — use dev / sandbox environments to learn and test, production for published agents.

Getting there

A pragmatic, low-risk rollout.

  1. 1

    Frame · Weeks 1–2

    Map who needs to run AI code — developers, makers, services — and match each to a sandbox surface. Confirm region & data-residency needs.

  2. 2

    Enable · Weeks 3–4

    Turn on the Copilot cloud-sandbox access policy and Copilot Studio code interpreter in a pilot environment. Stand up a Foundry project for code-first teams.

  3. 3

    Govern · Weeks 5–7

    Wire Intune/MDM policy, Azure RBAC, and audit-log routing. Define which environments allow execution and review the human-gate steps.

  4. 4

    Scale · Quarter 2+

    Make sandboxed execution the default path for agentic work; track usage, sessions, and cost; expand from pilots to the wider org.

Honest about trade-offs

Risks & mitigations.

Preview surface

Copilot sandboxes are public preview and may change. Pilot deliberately; avoid betting critical workflows on preview-only behavior yet.

No egress = limits

Locked-down networking blocks legitimate API calls too. Use Foundry's custom code interpreter or connectors when controlled egress is genuinely required.

Cost & sessions

Cloud sandboxes and Foundry sessions bill on top of usage; concurrent conversations spawn separate sessions. Monitor and consolidate.

Sovereign gaps

Copilot Studio code interpreter is public-cloud only today (no GCC / GCC High). Validate regional & sovereign coverage before committing regulated workloads.
